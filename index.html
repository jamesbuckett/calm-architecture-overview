A Structured Primer

CALM Architecture, made machine‑readable.

A five-module tutorial on the FINOS Common Architecture Language Model: from origin and motivation, through anatomy and controls, to a practical workflow you can run on a regulated platform.

By github.com/jamesbuckett @jamesbuckett in/jamesbuckett
Module 01 Origin & Motivation

Why a new architecture language, in 2025?

CALM did not appear in a vacuum. It is a deliberate response to a recurring failure mode in regulated industries — the gap between architecture diagrams on a wiki and architecture contracts a CI pipeline can enforce.

1.1

A short history

The Common Architecture Language Model began as an internal architecture-as-code initiative inside Morgan Stanley, where reference architectures had to satisfy not only engineering goals but also a tight lattice of internal control standards and external regulators. In the early 2020s, that team faced a familiar problem: every architecture document was a bespoke artefact, every review cycle was manual, and every drift between design and reality was discovered late.

In 2024 the work was contributed to FINOS — the Fintech Open Source Foundation, hosted under the Linux Foundation — and incubated inside the new Architecture as Code (AasC) Community of Interest. After a year of public iteration, CALM v1.0 was released in August 2025, with active contributions from Tier 1 banks and platform vendors. It is now one of the very few openly governed, machine-readable architecture standards designed with regulated environments in mind.

1.2

The problem CALM was designed to solve

Existing architecture artefacts each fall short on at least one axis that matters to a regulated platform team:

A

C4 Model

Brilliant for communicating architecture to humans. But the artefact is a diagram — there is no schema for a continuous-integration job to validate against, and no first-class place to attach a control.

B

ArchiMate / TOGAF

Comprehensive enterprise modelling, often locked behind heavyweight tooling and rarely loved by application engineers. Closer to a process framework than a developer-readable artefact.

C

UML

Formal but increasingly out of fashion in cloud-native shops. Component diagrams describe code structure better than they describe distributed systems and identity flows.

D

Plain Mermaid / draw.io

Quick, readable, and easy to commit to Git. But every team invents its own conventions; nothing is validated, and nothing carries semantic meaning a tool can act on.

E

Ad-hoc YAML / JSON

The home-grown "we'll just describe our architecture in YAML" approach. Different across every team, no shared schema, no shared patterns, no portability.

F

Terraform / Crossplane

These describe infrastructure, not architectural intent. A Terraform module knows that it provisions a load balancer; it does not know that the load balancer terminates an external trust boundary.

CALM is not a replacement for any of these. It is a complement: a machine-readable, schema-validated, control-aware source of truth that the others can render from, plug into, or reference.

1.3

Community and contributors

CALM is governed under FINOS, the only fintech-focused open source foundation operating with formal Linux Foundation governance. That matters: the spec is contributed to and consumed by institutions whose risk appetites usually preclude unsigned third-party tooling. The FINOS process — IP cleared, contributor-licence-agreement based, open meetings — is what makes CALM legally usable inside Tier 1 banks in the first place.

Active contributors and consumers include large global banks, the FINOS staff team, and a growing number of platform-engineering vendors. Pattern libraries from member firms are increasingly being published to the CALM Hub, the public reference catalogue maintained by the project.

The canonical sources are:

  • · github.com/finos/architecture-as-code — the monorepo containing the schema, CLI, hub UI, and reference patterns.
  • · npmjs.com/package/@finos/calm-cli — the official command-line tool.
  • · FINOS project page, accessed through finos.org under the AasC Community of Interest.
1.4

Where CALM fits in the Architecture-as-Code movement

The Architecture-as-Code movement asks the same question that Infrastructure-as-Code answered a decade ago: what would happen if our most authoritative architectural decisions lived in a Git repository, were peer-reviewed in pull requests, and were enforced by CI? CALM provides one of the missing pieces — a portable, vendor-neutral file format with the semantic richness to host controls and patterns, not just shapes and lines.

Crucially, CALM is positioned upstream of IaC. Terraform, Crossplane, Helm and Kustomize describe how a system is realised; CALM describes what the system is supposed to be in the first place. The two are complementary, and both belong in the same source-controlled repository.

Key Takeaways · Module 01
  • CALM was created at Morgan Stanley, donated to FINOS, and reached v1.0 in August 2025. It is governed in the open under Linux Foundation rules.
  • Its purpose is to fill the gap between visual architecture (C4, ArchiMate) and infrastructure-as-code — providing a schema-validated, control-aware source of truth.
  • It complements rather than replaces existing tools. C4 diagrams can be rendered from a CALM document; Terraform can be informed by one.
  • It is purpose-built for regulated industries: open governance, controls as first-class citizens, and reusable patterns that can encode reference architectures.

Module 02 Conceptual Overview

Six primitives, one schema, and a single hard distinction.

CALM is small. The whole language can be sketched on the back of an envelope. Most of its power comes from a clean separation between Patterns and Architectures — and from making Controls a first-class citizen.

2.1

The six core building blocks

Every CALM document is composed from six primitives. If you can describe a system using these six, you have a valid CALM model.

01

Nodes

The things in your architecture. A node may be a service, a database, a network segment, a person/actor, an external system, or a logical grouping.

02

Relationships

Typed edges between nodes. CALM knows the difference between connects, deployed-in, composed-of and interacts.

03

Interfaces

How a node exposes itself: a host:port pair, an HTTP route, a gRPC service, a queue topic. Interfaces are what relationships actually attach to.

04

Controls

Policy, security, and compliance requirements bound to a node or relationship. Controls reference external requirement catalogues and concrete implementations.

05

Flows

A named, sequenced traversal across multiple relationships. Use flows to describe a sign-in journey, a payment path, or any cross-component use case.

06

Metadata

Extensible key/value extension points — for ownership, environment tags, classification, regulatory scope, or anything your governance model needs.

2.2

The schema, and why JSON Schema

The CALM specification is published as a set of JSON Schema documents. This is a deliberate choice. JSON Schema is mature, language-agnostic, mechanically validatable in any modern toolchain, and extensible — a CALM-Hub pattern can extend the base schema to add stricter constraints (e.g. "every node must declare an owner").

Concretely, every CALM document begins with a $schema reference. The CLI uses that reference to fetch and apply the appropriate validator. This matters in CI: a malformed CALM document fails the build the same way malformed Terraform does.

2.3

Pattern vs Architecture — the one distinction that matters

This is the single most important conceptual move in CALM.

  • · A Pattern is a template. It declares the structure that any conforming architecture must follow, the controls that must be present, and the placeholders that an architect must fill in. Think of it as the regulator-approved reference architecture.
  • · An Architecture is an instance. It fills in the placeholders, names concrete services, and adds project-specific detail. It must validate against the Pattern it claims to implement.

If you have used JSON Schema before, the analogy is exact: Pattern is to Architecture as JSON Schema is to a JSON document. The CALM CLI's validate command checks the Architecture against both the base CALM schema and the Pattern it references.

2.4

How CALM relates to C4, TOGAF, and IaC

Treat CALM as the source of truth, and treat the others as either views into it or downstream consumers of it:

  • · C4 — render Context, Container and Component diagrams from the CALM document. The CALM document is the spec; the C4 diagrams are the documentation.
  • · TOGAF — TOGAF tells you how to do enterprise architecture; it does not give you a file format. CALM is the file format your TOGAF process produces.
  • · Terraform / Crossplane — these realise the architecture. A CALM document can pin which provider modules are acceptable; a Terraform module can be tagged with the CALM node it implements.
  • · OPA / Kyverno / policy-as-code — controls in CALM reference the policies that enforce them at runtime, closing the loop between design-time intent and runtime enforcement.

Key Takeaways · Module 02
  • CALM has six primitives: Nodes, Relationships, Interfaces, Controls, Flows, Metadata.
  • The format is JSON validated by JSON Schema — every document declares a $schema.
  • The crucial conceptual move is Pattern (template) vs Architecture (instance); a pattern is to an architecture what a JSON Schema is to a JSON document.
  • CALM does not replace C4 or Terraform; it sits upstream of them as the source of truth, and renders into them as views.
  • Compared to ArchiMate, CALM is friendlier to engineers and to Git diffs; compared to plain Mermaid, it is enforceable.

Module 03 Anatomy of a CALM Document

From the smallest valid file to a real three-tier system.

In this module we evolve a CALM document from the bare minimum into a realistic, validating, render-ready architecture for a customer-facing web application — load balancer, web tier, API tier, database, and external identity provider.

3.1

The minimal valid CALM document

The smallest document the CLI will accept needs four things: a schema reference, an identity for the document itself, and the two collections that everything else hangs off — nodes and relationships. Both may be empty.

hello-world.arch.json CALM · JSON 
{
  "$schema": "https://calm.finos.org/release/1.0/meta/calm.json",
  "name": "Hello World",
  "description": "The smallest CALM document that validates.",
  "nodes": [],
  "relationships": []
}

Run calm validate -a hello-world.arch.json against this and you will see 0 errors. It is empty, but it is valid.

Note · schema URL

Schema URLs evolve between CALM releases. Always reference the URL printed by the CLI for your installed version (calm --version) rather than copying one from a tutorial. The path shown above is illustrative; verify against the FINOS repository.

3.2

Adding nodes and relationships

We now describe a real system: a customer-facing web application with a public load balancer, a web tier, an API tier, a database, and a federated identity provider. Each node has a unique id, a type, and a human-readable name. Relationships reference nodes by id and declare what kind of edge they are.

webapp.arch.json — nodes & relationships CALM · JSON 
{
  "$schema": "https://calm.finos.org/release/1.0/meta/calm.json",
  "name": "Customer Web App",
  "description": "Public-facing 3-tier web application.",
  "nodes": [
    { "unique-id": "customer", "node-type": "actor",    "name": "Customer" },
    { "unique-id": "alb",      "node-type": "network",  "name": "Application LB" },
    { "unique-id": "web",      "node-type": "service",  "name": "Web Tier (SSR)" },
    { "unique-id": "api",      "node-type": "service",  "name": "API Tier" },
    { "unique-id": "db",       "node-type": "database", "name": "PostgreSQL" },
    { "unique-id": "idp",      "node-type": "system",   "name": "OIDC Provider" }
  ],
  "relationships": [
    {
      "unique-id": "customer-to-alb",
      "relationship-type": { "connects": { "source": { "node": "customer" },
                                          "destination": { "node": "alb" } } },
      "protocol": "HTTPS"
    },
    { "unique-id": "alb-to-web",
      "relationship-type": { "connects": { "source": { "node": "alb" },
                                          "destination": { "node": "web" } } },
      "protocol": "HTTPS" },
    { "unique-id": "web-to-api",
      "relationship-type": { "connects": { "source": { "node": "web" },
                                          "destination": { "node": "api" } } },
      "protocol": "HTTPS / mTLS" },
    { "unique-id": "api-to-db",
      "relationship-type": { "connects": { "source": { "node": "api" },
                                          "destination": { "node": "db" } } },
      "protocol": "PostgreSQL/TLS" },
    { "unique-id": "web-to-idp",
      "relationship-type": { "connects": { "source": { "node": "web" },
                                          "destination": { "node": "idp" } } },
      "protocol": "OIDC / HTTPS" }
  ]
}

3.4

Adding interfaces, controls and a flow

Now we layer the remaining primitives. Interfaces give relationships somewhere to attach. Controls express a policy requirement. Flows describe the customer sign-in journey end-to-end.

webapp.arch.json — interfaces, controls, flows CALM · JSON (excerpt) 
"interfaces": [
  { "unique-id": "alb-https", "node": "alb",
    "interface-type": "https", "port": 443 },
  { "unique-id": "api-grpc", "node": "api",
    "interface-type": "grpc", "port": 8443 }
],
"controls": [
  { "unique-id": "mtls-internal",
    "description": "All east-west calls must use mTLS with SPIFFE SVIDs.",
    "requirement-url": "https://nvlpubs.nist.gov/SP800-207",
    "applies-to": [ "web-to-api", "api-to-db" ] }
],
"flows": [
  { "unique-id": "customer-sign-in",
    "description": "OIDC sign-in for an authenticated session.",
    "transitions": [
      { "sequence": 1, "relationship": "customer-to-alb" },
      { "sequence": 2, "relationship": "alb-to-web" },
      { "sequence": 3, "relationship": "web-to-idp" },
      { "sequence": 4, "relationship": "web-to-api" }
    ] }
]
3.5

Validation in practice

The CLI validates an architecture against the base CALM schema and, optionally, against a Pattern. Failures are structured (exit code, file/line) and slot directly into a CI job.

~/repo $ calm validate -a webapp.arch.json SHELL 
// success
→ Loaded webapp.arch.json
→ Validating against https://calm.finos.org/release/1.0/meta/calm.json
→ 6 nodes · 5 relationships · 2 interfaces · 1 control · 1 flow
→ PASS · 0 errors · 0 warnings

// failure example — referencing a non-existent node
→ ERROR  /relationships/3/relationship-type/connects/destination/node
   value "db-replica" does not match any node unique-id.
→ FAIL · 1 error
Key Takeaways · Module 03
  • A valid CALM file needs only a $schema, a name, and (possibly empty) nodes and relationships arrays.
  • Nodes have a unique-id, a node-type, and a name; relationships are typed and refer to nodes by id.
  • Interfaces, controls, flows and metadata are additive — you can grow a document gradually without breaking validation.
  • The CLI validates against the base schema and against any referenced Pattern, exiting non-zero on failure for clean CI integration.

Module 04 Patterns, Controls, Governance

Where CALM stops being a diagram tool and becomes a control plane.

For regulated industries the interesting question is not “can I draw my system?”, it is “can I prove that my system meets a recognised control framework, every time it changes?”. This module shows how Patterns and Controls answer that question.

4.1

Patterns as opinionated reference architectures

A Pattern is a reusable, opinionated CALM document. It captures the architectural decisions a regulated firm has already made — the ones that should not be re-litigated on every project — and exposes the deliberately-variable bits as placeholders.

Typical things a Pattern locks down:

  • · Required node types — every customer-facing pattern must include a WAF and an IdP.
  • · Required controls — mTLS on east-west, TLS 1.2+ on north-south, key rotation under 90 days.
  • · Forbidden topologies — no direct internet exposure of databases, no shared service accounts.
  • · Mandatory metadata — every node must declare owner, data-classification, regulatory-scope.

An architect implementing the pattern is free to choose specific service implementations (Postgres or Aurora? Envoy or Istio?) but cannot opt out of the controls.

4.2

Controls as first-class citizens

A Control in CALM is a structured object: a description, a reference to an external requirement (e.g. a NIST Special Publication, a PCI-DSS sub-requirement, an internal control standard), and a list of nodes or relationships it applies to. Controls travel with the architecture and are evaluated by the validator.

The structural pattern is consistent: requirement-document → control-detail → applies-to.

controls fragment — illustrative CALM · JSON 
"controls": [
  {
    "unique-id": "zt-explicit-trust",
    "description": "All access decisions must be made on a per-request basis.",
    "requirement": {
      "framework": "NIST SP 800-207",
      "requirement-id": "3.1",
      "requirement-url": "https://nvlpubs.nist.gov/nistpubs/SpecialPublications/NIST.SP.800-207.pdf"
    },
    "control-detail": {
      "implementation": "OPA-Envoy ext-authz at every PEP",
      "evidence-url": "git://platform/policies/zt-explicit"
    },
    "applies-to": [ "web-to-api", "api-to-db" ]
  },
  {
    "unique-id": "zt-workload-identity",
    "description": "Every workload must present a SPIFFE SVID, not a long-lived secret.",
    "requirement": { "framework": "NIST SP 800-207", "requirement-id": "3.4" },
    "control-detail": { "implementation": "SPIRE-issued SVIDs, 1-hour TTL" },
    "applies-to": [ "web", "api" ]
  }
]
4.3

Drift, conformance, and ADR generation

Once your architecture is in CALM, three governance behaviours become straightforward:

  • · Conformance — every pull request runs the validator. If the change violates the Pattern (e.g. removes a required control), CI blocks the merge.
  • · Drift detection — periodic jobs compare the deployed state (read from cloud APIs / Kubernetes / service mesh) against the CALM document. Drift opens a ticket; the source of truth never silently changes.
  • · ADR generation — material changes to the CALM document trigger an Architecture Decision Record in docs/adr/, automatically pre-populated with the diff, the affected controls, and the affected pattern.

The result is that architectural change in a regulated environment becomes auditable by construction, not by retrospective evidence-gathering.

4.4

Worked example — a Zero Trust pattern

Below is a sketch of a Zero Trust web application pattern, with controls mapped to NIST SP 800-207. In the rendered diagram each control is shown as a small badge attached to the relationship it applies to.

When a project team uses this pattern as the basis for a new architecture, the validator enforces that every required control is applied. Removing a control (say, downgrading from mTLS to one-way TLS for an internal call) causes the build to fail with an explicit message and the offending diff.

Key Takeaways · Module 04
  • A Pattern is your firm's opinionated reference architecture, expressed as a CALM document with required controls and placeholders.
  • A Control is a structured object linking a requirement framework to a specific implementation, scoped to particular nodes or relationships.
  • Controls enable conformance, drift detection, and auto-generated ADRs — making architectural change auditable by construction.
  • NIST SP 800-207 (Zero Trust) is a natural early target; the same approach scales to PCI-DSS 4.0, DORA, MAS TRM, APRA CPS 234, etc.

Module 05 Practical Use & Tooling

The CLI, the workflow, the roles, and a starter exercise.

CALM is only useful if it lives where developers and architects already work — in Git, in pull requests, in CI. This module covers the everyday mechanics, then ends with a hands-on exercise you can complete in your home lab.

5.1

Installing and using the CALM CLI

The CLI is published as an npm package, @finos/calm-cli. The most useful sub-commands are validate, generate, and visualize.

~ $ install & use SHELL 
# install (use a user-owned npm prefix; do not sudo)
npm install -g @finos/calm-cli
calm --version

# validate an architecture against the base schema
calm validate -a webapp.arch.json

# validate against a specific pattern
calm validate -p zero-trust-web.pattern.json -a webapp.arch.json

# generate a starter architecture from a pattern (fills placeholders)
calm generate -p zero-trust-web.pattern.json -o webapp.arch.json

# render a diagram (SVG) from an architecture
calm visualize -a webapp.arch.json -o webapp.svg

# render a C4-style container diagram
calm visualize -a webapp.arch.json --view container -o webapp-container.svg
Note · exact flags

CLI flags evolve between releases; treat the snippet above as illustrative. Always confirm with calm --help on your installed version, and pin a specific CLI version in CI for reproducibility.

5.3

Roles — who does what?

Adoption fails when CALM is treated as “the architect's file”. Three roles share the artefact, each with a distinct responsibility.

A

Architect

Owns the Pattern. Decides which controls are required, which topologies are forbidden, and which placeholders projects can fill in.

B

Platform PM / Engineer

Owns the validator integration in CI, the CLI version pinning, the dashboard rendering, and the tooling that turns CALM diffs into ADRs.

C

Application engineer

Owns the Architecture for their service. Edits the CALM document in pull requests just like any other source file. Reviews the rendered diagram in code review.

5.4

Adoption signals to look for

FINOS members publishing patterns, vendors adding CALM export to their tools, and the CALM Hub catalogue growing are all positive adoption signals. In a Tier 1 firm, more practical signals include:

  • · Architecture review boards reviewing CALM diffs in PRs rather than slide decks.
  • · Internal control standards being expressed once, in CALM, and consumed everywhere.
  • · Drift jobs reporting against a single source of truth — the CALM document — rather than reconciling several inventories.
  • · Auditors asking for a CALM export instead of a Visio file.
5.5

Starter exercise — your home lab

The fastest way to learn CALM is to model something you already operate. Try this in roughly an hour:

Exercise · Model a small service
  • Install the CLI: npm install -g @finos/calm-cli in a user-owned npm prefix. Confirm with calm --version.
  • Pick a small service you operate or know well — a personal blog, a side-project API, a homelab Kubernetes cluster.
  • Create homelab.arch.json: nodes for your front door, your service, your database, any external integrations.
  • Add at least three relationships and one flow representing the primary user journey.
  • Run calm validate -a homelab.arch.json until it passes.
  • Add a single control — for example, “all external traffic must be HTTPS-only” — and apply it to the relevant relationship.
  • Render: calm visualize -a homelab.arch.json -o homelab.svg. Compare the rendered diagram against the C4 diagram you would have drawn by hand.
  • Commit the file and the rendered SVG to a Git repository. You now have an architecture-as-code artefact you can review in a pull request.
Key Takeaways · Module 05
  • The CLI lives at @finos/calm-cli on npm. Its three core commands are validate, generate and visualize.
  • Treat CALM documents as source code — committed, reviewed, validated in CI. The validator is the same gate your tests are.
  • Three distinct roles share the artefact: architect (pattern), platform team (tooling), application engineer (architecture).
  • Start small: a single service modelled in your home lab is enough to internalise the workflow.

Common anti-patterns

  • Treating CALM as a diagram tool — the diagram is a view, the JSON is the spec.
  • Letting the architecture document drift from runtime — wire up drift detection from day one.
  • Creating a single mega-pattern. Prefer small, focused patterns (e.g. zero-trust-web, batch-job).
  • Hand-rolling JSON without the CLI. Validate locally before pushing.
  • Copy-pasting controls between docs. Express controls once, in patterns; apply them by reference.
